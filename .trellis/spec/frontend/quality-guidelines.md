# Quality Guidelines

> Code quality standards for frontend development.

## Required Checks

For non-trivial frontend changes, run the relevant available checks:

- `bun run --cwd apps/web typecheck`.
- `bun run --cwd apps/web lint`.
- `bun run --cwd apps/web format:check`.
- `bun run --cwd apps/web build`.
- Unit/component tests for touched logic.
- Playwright or equivalent E2E coverage for critical flows when implemented.

Report checks that were not run and why.

## Package Manager

- Use Bun for frontend dependency management and scripts.
- Frontend commands should run from the repository root with `bun run --cwd apps/web <script>`.
- Do not commit `package-lock.json`, `yarn.lock`, or `pnpm-lock.yaml`.
- Commit the Bun lockfile generated by the project Bun version.
- Pin the Bun version in `packageManager` once the root or frontend `package.json` is created.

## Lint And Formatting Baseline

Use this baseline for `apps/web`:

- ESLint Flat Config.
- Prettier.
- TypeScript strict.
- `eslint-plugin-react`.
- `eslint-plugin-react-hooks`.
- `eslint-plugin-jsx-a11y`.
- `eslint-plugin-simple-import-sort`.
- `eslint-config-prettier`.

Rule posture:

- `react-hooks/rules-of-hooks`: error.
- `react-hooks/exhaustive-deps`: warning initially.
- `@typescript-eslint/consistent-type-imports`: error.
- `simple-import-sort/imports`: error.
- `simple-import-sort/exports`: error.
- `@typescript-eslint/no-explicit-any`: warning initially, error after the codebase stabilizes.
- `no-console`: warning, allowing `warn` and `error`.

## Git Workflow

- Branch, PR, commit, and merge policy comes from `CONTRIBUTING.md`.
- Current frontend work follows the repository fork + PR flow into `upstream/develop`.
- Create frontend feature branches from the latest `upstream/develop`.
- Push feature branches to the developer fork (`origin`).
- Open PRs from the fork feature branch into `Sakayori-Iroha-168/Software_Teamwork:develop`.
- Do not use `frontdev` in commands, docs, CI, or PR targets.
- Do not target `frontend-dev` unless `CONTRIBUTING.md` is updated to enable that branch.
- Use Conventional Commits for frontend commits.

## Testing Expectations

- Use Vitest for pure functions, hooks, schema transforms, permission helpers, and stream parsers.
- Use React Testing Library for feature components with meaningful UI states.
- Use Playwright for login, document upload flow, chat streaming, report generation wizard, and download flows once those screens exist.
- Mock network at the API boundary. Do not test generated OpenAPI internals unless generated code is customized.

## UI State Coverage

Every data-driven page should handle:

- Loading.
- Empty.
- Error.
- Permission denied.
- Partial/progress states for long-running tasks.
- Success and destructive-action confirmation.

Streaming UIs should handle:

- Connecting.
- Streaming.
- Cancellation.
- Error with partial content.
- Done.

## Forbidden Patterns

- Direct DOM manipulation for normal React UI.
- `dangerouslySetInnerHTML` for AI or document content unless sanitized and reviewed.
- One-off fetch calls in components when a TanStack Query hook should own server state.
- Silent mutation failures.
- Components that assume admin permissions without guard checks.
- Upload flows without file type, size, progress, and error states.
- Rich text/report editing without a structured save format.

## Required Patterns

- Use generated API types where backend contracts exist.
- Use Zod for user input and runtime validation at trust boundaries.
- Use `AbortController` for cancellable streams and uploads when possible.
- Keep destructive actions behind confirmation.
- Keep permission checks close to route definitions and action rendering.
- Use shared empty/error/loading components for consistent management UI.

## Review Checklist

- Does this code match the recommended stack and directory boundaries?
- Does the PR target match `CONTRIBUTING.md`? Current default: `develop`.
- Were frontend commands run with Bun?
- Is server state owned by TanStack Query?
- Are route params/search params typed and validated?
- Are forms backed by React Hook Form and Zod?
- Are loading, empty, error, and permission states present?
- Are streaming/upload/long-task paths cancellable or recoverable?
- Are backend authorization and frontend visibility treated separately?
- Are user-visible labels concise and domain-appropriate?
