# 整体需求与 API 契约分析：智能知识管理与报告生成系统

## Goal

基于四份需求说明书和 `docs/` 既有材料，产出一份统一的整体需求分析文档，并按知识管理、智能问答、报告生成三个业务模块分别产出 API 契约文档。目标是尽可能覆盖四份原始需求，同时保持模块解耦，明确 PostgreSQL、Qdrant、MinIO、Redis 等基础设施职责，且对不明确事项优先标记为待确认。

## What I already know

- 当前已有四份原始需求说明书，并已转换为 Markdown：
  - `知识管理需求说明书(1).md`
  - `智能问答需求说明书(1).md`
  - `报告生成需求说明书(1).md`
  - `技术监督辅助平台需求说明书(1).md`
- `docs/` 下已有旧版模块说明：`docs/knowledge_management_system.md`、`docs/smart_quote_system.md`、`docs/report_generation_system.md`、`docs/system.md`。
- 项目 README 和 backend spec 已明确目标架构：Go 微服务、Gateway/Auth/File/Knowledge/QA/Document 服务，HTTP/REST 通信，PostgreSQL、Redis、Qdrant、MinIO 作为基础设施。
- 四份文档存在明显交叉：知识管理提供文档、切片、向量化与检索底座；智能问答复用知识库和检索能力；报告生成复用专业素材、模板和模型能力；技术监督辅助平台是三类能力的综合上位描述。
- 需求分析应站在整体系统视角，而不是只审查单一“知识管理智能问答智能分析”文档。
- 文档应面向后续产品拆解、架构设计和开发排期，避免堆砌实现细节。

## Confirmed Decisions

- 以“智能知识管理与报告生成系统”为整体产品对象进行需求分析。
- 知识管理、智能问答、报告生成是核心业务模块；技术监督辅助平台需求说明书作为整体平台视角的主参考。
- 报告支撑材料需要作为报告生成复用的专业业务文档来定义，避免误解为 UI 素材或普通附件。
- 文档管理属于知识库管理内部能力，文档必须归属于某个知识库。
- 整体需求分析先聚焦业务目标、模块关系、用户角色、核心流程、非功能要求和待确认问题，不直接进入代码实现。
- API 契约按三个业务模块拆分：知识管理、智能问答、报告生成；用户认证、文件对象存储、模型配置、统计监控作为横向能力在契约中引用但不混入业务职责。
- Qdrant 只由知识管理模块/knowledge 服务拥有写入和索引维护；智能问答和报告生成通过检索 API 复用知识能力，不直接读写 Qdrant。

## Requirements

- 阅读并整合四份 Markdown 需求说明书，识别重复、交叉和冲突的需求。
- 产出整体需求分析文档，建议目标路径为 `docs/需求分析/整体需求分析.md`。
- 产出三份 API 契约文档：
  - `docs/接口契约/知识管理-api契约.md`
  - `docs/接口契约/智能问答-api契约.md`
  - `docs/接口契约/报告生成-api契约.md`
- 产出集中待确认决策清单：`docs/需求分析/待确认决策清单.md`
- 明确系统整体定位、目标用户、核心业务价值和验收关注点。
- 梳理核心模块：
  - 知识管理
  - 智能问答
  - 报告生成
  - 技术监督辅助平台综合能力
  - 用户、权限、配置、统计监控等横向能力
- 明确模块间依赖关系，例如“知识管理是问答和报告生成的数据底座”。
- 汇总用户角色、关键业务流程、核心数据对象和主要 API/页面能力边界。
- 标注 MVP 优先级、加分项、暂缓项和待确认问题。
- 保留需求来源映射，便于后续追溯到四份原始说明书。
- API 契约需要说明 endpoint、请求/响应、状态流转、权限、错误码、基础设施归属和模块边界。
- 不明确项不得擅自写成已确认方案，应列入“待确认问题”。

## Out of Scope

- 不实现代码。
- 不做前端页面实现。
- 不做后端 API、数据库或模型调用实现。
- 不重写四份原始需求说明书。
- 不做完整 UI 原型和交互设计。
- 不提交代码，除非用户另行确认。
- OpenAPI YAML 仅作为接口契约草稿，不实现服务内 `services/*/api/openapi.yaml` 文件，不接入运行时 Swagger UI。
