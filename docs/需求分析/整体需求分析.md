# 整体需求分析：智能知识管理与报告生成系统

## 1. 文档目标

本文基于根目录四份需求说明书和 `docs/` 下既有模块文档，整理一份统一、可拆解、可追溯的整体需求分析。目标不是简单合并原文，而是把重复需求、交叉需求和模块依赖梳理成后续产品设计、后端服务拆分、API 契约和开发排期可以直接使用的上位需求。

本文遵循两个原则：

- 尽可能覆盖四份原始需求说明书提出的能力，避免遗漏核心考核点。
- 尽可能保持模块解耦，知识管理、智能问答、报告生成各自有清晰边界，通过稳定 API 协作，而不是直接共享内部存储或实现细节。

## 3. 产品定位

系统定位为面向电力行业的智能知识管理与报告生成平台。平台以行业文档、技术资料、报告材料和知识库为基础，提供从知识沉淀、知识检索、智能问答到专业报告生成的闭环能力。

整体业务链路是：

```text
文件上传
  -> 文档解析
  -> 分段切片
  -> 向量化索引
  -> 知识检索
  -> 智能问答 / 报告生成
  -> 引用溯源 / DOCX 导出
```

核心价值：

- 把电力行业资料沉淀为可管理、可检索、可复用的知识资产。
- 通过 RAG 提升问答准确率，减少用户在规程、报告、术语资料中的人工查找成本。
- 通过固定报告类型和模板驱动生成专业报告，提升迎峰度夏检查报告、煤库存审计报告等场景的编写效率。
- 为管理员提供模型配置、检索测试、模板管理、统计监控等运营能力。

核心考核点：

- 智能问答准确率。
- 报告生成完成度和专业质量。
- 文档处理、检索召回、引用溯源、导出样式的一致性和稳定性。

## 4. 用户角色与权限

| 角色 | 定位 | 主要权限 |
| --- | --- | --- |
| 标准用户 | 使用系统完成检索、问答、报告生成 | 查询有权限访问的知识库，发起问答，生成和下载自己的报告 |
| 管理员 | 管理知识资产和业务配置 | 管理知识库、文档、报告模板、报告支撑材料、模型配置、检索测试、统计看板 |
| 超级管理员 | 系统最高权限角色 | 拥有全部管理能力，负责用户、角色、全局配置和敏感操作 |

权限原则：

- 所有 API 请求必须经过身份认证。
- 首期认证方式统一为 Bearer Token/JWT，由 `gateway` 或对应服务校验 `Authorization: Bearer <accessToken>`。
- 管理功能仅限管理员和超级管理员访问。
- 用户只能检索和下载自己有权限访问的知识库、文档、引用原文和报告文件。
- 意图路由需要结合权限判断。例如用户没有报告生成权限时，即使识别出报告生成意图，也不能进入报告生成流程。
- 首期数据权限采用角色级 RBAC，暂不引入部门、组织、电厂、专业等多维权限。
- 删除知识库、文档、模板、报告记录等操作需要二次确认；首期不建设完整审计日志和审计查询服务，但配置变更、任务失败、删除结果应保留基础排查字段。

## 5. 模块边界

### 5.1 服务与模块关系

项目目标架构采用微服务拆分：

| 服务 | 业务责任 |
| --- | --- |
| `gateway` | 统一 API 入口，路由、鉴权上下文传递、聚合接口、跨服务协调 |
| `auth` | 用户、角色、权限、登录认证、会话或令牌管理 |
| `file` | 上传文件、对象存储、文件元数据、下载授权、预签名 URL |
| `knowledge` | 知识库、文档处理、切片、向量化、Qdrant 索引、检索协调 |
| `qa` | 会话、消息、意图识别、RAG 编排、答案生成、引用展示 |
| `document` | 报告模板、报告任务、大纲、章节内容、DOCX 导出、报告记录 |
| `ai-gateway` | 统一封装 LLM、Embedding、Rerank 等模型调用，向业务服务提供 OpenAI-compatible API |

业务模块与服务的关系：

| 业务模块 | 主责服务 | 依赖服务 |
| --- | --- | --- |
| 知识管理 | `knowledge` | `file`、`auth` |
| 智能问答 | `qa` | `knowledge`、`auth`、`ai-gateway`、可选 `file` |
| 报告生成 | `document` | `knowledge`、`file`、`auth`、`ai-gateway` |
| 横向管理 | `gateway`、`auth`、各业务服务 | 所有服务 |

### 5.2 解耦原则

- `knowledge` 是 Qdrant 的唯一业务所有者，负责向量集合、payload 结构、索引写入、重建和检索策略。
- `qa` 不直接读写 Qdrant，只通过 `knowledge` 的检索 API 获取候选片段和引用元数据。
- `document` 不直接绕过 `knowledge` 做向量检索；报告生成需要业务材料时，通过知识检索或报告支撑材料接口获取。
- `file` 是 MinIO 对象存储边界，业务服务只保存文件引用和业务元数据，不直接把对象 key 当作权限依据。
- `file` service 是后端与文件上传、下载授权和 MinIO 对象存储沟通的中间件，业务模块不得绕过它直接暴露 object key。
- PostgreSQL 保存业务真相；Qdrant 保存检索向量和最小检索 payload；MinIO 保存原始文件、解析产物和生成文档；Redis 用于队列、短期任务状态、缓存和 SSE 辅助状态，不作为长期业务真相。
- 模型调用通过 `ai-gateway` 统一封装为 OpenAI-compatible API；业务服务通过配置中的 `baseURL`、`apiKey` 和模型名调用，不各自实现供应商适配层。

## 6. 基础设施职责

| 基础设施 | 职责 | 不能承担的职责 |
| --- | --- | --- |
| PostgreSQL | 用户、角色、权限、知识库、文档元数据、处理状态、切片元数据、会话、消息、报告任务、模板、统计聚合所需业务数据 | 不存储大体积原始文件，不替代向量检索 |
| Qdrant | 文档切片向量、检索 payload、相似度查询 | 不保存完整业务真相，不由 QA 或 Document 直接写入 |
| MinIO | 原始上传文件、模板文件、生成 DOCX、较大的解析文本或中间产物 | 不作为权限判断唯一依据 |
| Redis | 异步任务队列、会话缓存、短期任务状态、热点配置缓存、SSE 连接辅助状态 | 不保存必须长期追溯的业务状态 |
| AI Gateway | 统一 LLM、Embedding、Rerank 等模型调用，提供 OpenAI-compatible API | 不保存业务真相，不把模型密钥透传给前端 |

基础设施决策：

- 首期异步任务采用 Redis 队列 + PostgreSQL 持久化状态。
- OCR 和文档解析首期使用外部 HTTP 解析服务跑通 pipeline；解析服务通过 `parser.baseUrl`、`apiKey`、`timeoutSeconds`、`maxConcurrency` 配置，任务失败最多自动重试 3 次。
- LLM、Embedding、Rerank 等模型调用统一经 `ai-gateway`，各业务服务首期只需按 OpenAI-compatible API 调用单一可用供应商。
- MinIO bucket 首期按 `source-files`、`templates`、`generated-reports` 三类拆分；具体 bucket 名可由部署环境变量配置。

## 7. 核心数据对象

### 7.1 用户与权限

- `User`：用户账号、显示名、状态、创建时间。
- `Role`：标准用户、管理员、超级管理员。
- `Permission`：知识管理、知识问答、报告生成、系统配置等角色权限；数据分析权限仅作为后续扩展预留。
- `AccessToken`：Bearer Token/JWT 登录凭证，由 `auth` 签发并经 `gateway` 校验。

### 7.2 知识管理

- `KnowledgeBase`：知识库，包含名称、描述、文档类型、可见范围、分段策略、检索策略、创建人。
- `KnowledgeDocument`：知识库内文档，必须归属某个知识库，包含文件引用、标签、状态、错误信息。
- `DocumentChunk`：文档切片，包含章节路径、切片类型、文本摘要、向量索引引用、序号和元数据。
- `ProcessingJob`：文档解析、切片、向量化、重处理任务；PostgreSQL 保存权威状态、失败原因、尝试次数和最近 10 次尝试摘要。
- `RetrievalProfile`：Top K、相似度阈值、重排序阈值、是否启用重排序等检索参数。
- `ReportSupportMaterial`：报告支撑材料，是报告生成复用的专业业务文档，不是 UI 素材或普通附件；可与知识库文档共用处理和检索规则。

### 7.3 智能问答

- `Conversation`：用户会话，支持创建、切换、删除和历史恢复。
- `Message`：用户问题、助手回答、意图、生成状态、token 或耗时统计。
- `Citation`：引用片段，关联切片、来源文档、章节路径、相关性分数。
- `ReasoningStep`：面向用户展示的处理链路或思考步骤。为避免泄露模型内部隐私，建议展示“处理过程摘要”，而不是原始 chain-of-thought。
- `RetrievalTest`：管理员检索召回测试记录。

### 7.4 报告生成

- `ReportTemplate`：报告模板，包含报告类型、文件引用、样式配置、章节结构、模板状态。
- `ReportSupportMaterial`：报告支撑材料，可按类型和标签组合筛选。
- `Report`：报告记录，包含名称、类型、专业、电厂、年份、状态、所有者。
- `ReportOutline`：多级大纲，支持章节排序、删除、自动重新编号。
- `ReportSection`：章节内容，包含章节类型、生成状态、正文、表格或图片引用。
- `ReportExport`：导出文件记录，关联 DOCX 对象、样式版本和生成时间。

## 8. 核心业务流程

### 8.1 知识入库流程

1. 用户或管理员创建知识库，配置文档类型、分段策略和检索策略。
2. 用户上传文档，文件进入 MinIO，文件元数据进入 PostgreSQL。
3. `knowledge` 创建处理任务，状态从 `uploaded` 流转到 `parsing`。
4. 文档解析器把文件转换为文本或结构化内容；较大解析产物可落 MinIO。
5. 系统按知识库分段策略生成切片，并写入 PostgreSQL。
6. 嵌入模型生成向量，`knowledge` 写入 Qdrant。
7. 文档状态变为 `ready`；失败时记录错误原因并支持重试。
8. 当知识库分段策略或检索策略变更时，系统触发就绪文档的后台重处理。

### 8.2 知识检索流程

1. 用户选择一个或多个有权限访问的知识库。
2. 用户输入关键词或自然语言问题，并可配置标签过滤、Top K、阈值等参数。
3. `knowledge` 执行语义向量检索，必要时调用重排序模型。
4. 系统返回命中文档、章节路径、相关度分数和内容预览。
5. 检索结果可被前台检索页面、智能问答、报告生成复用。

### 8.3 智能问答流程

1. 用户创建或选择会话并提交问题。
2. `qa` 识别意图，判断是知识问答、一般对话、报告生成还是预留的数据分析意图。
3. 对知识问答意图，`qa` 调用 `knowledge` 检索 API 获取候选片段；对数据分析意图本期返回暂不支持或引导到后续能力。
4. `qa` 组织上下文，调用 LLM 生成答案。
5. 支持 SSE 流式输出，也支持非流式输出。
6. 回答完成后保存消息、引用、处理过程摘要和统计数据。
7. 用户可查看引用详情、原文片段、来源文档、章节路径、相关性分数，并在权限允许时下载原始文件。

### 8.4 报告生成流程

1. 用户选择报告类型，例如迎峰度夏检查报告或煤库存审计报告。
2. 用户输入报告主题、专业、电厂、年份等基础信息。
3. 系统根据固定模板或 AI 生成多级大纲。
4. 用户编辑大纲，系统自动重新编号。
5. 系统逐章节流式生成内容，展示总体进度和章节状态。
6. 章节内容支持用户在线编辑，必要时可重新生成单章内容。
7. 报告正文、表格、图片等内容保存为结构化数据。
8. 用户导出 DOCX，系统按统一样式体系生成文件并保存到 MinIO。
9. 报告记录支持查看、删除、重新生成 Word、重新下载已生成文件。

## 9. 三模块功能需求分解

本节按知识管理、智能问答、报告生成三个业务模块拆分。每个小功能都说明三件事：

- 功能用于实现什么：面向用户或系统的业务目标。
- 需要什么数据/API：该功能依赖的输入数据、内部服务或外部 API。
- 对外提供什么数据/API：该功能对前端、其他模块或管理端暴露的能力。

### 9.1 知识管理模块

知识管理是系统的数据底座，负责把原始文件沉淀为可检索、可引用、可复用的知识资产。

### 9.2 智能问答模块

## 10. API 契约拆分

本任务同步产出三份 API 契约：

- 知识管理：[知识管理-api契约.md](../接口契约/知识管理-api契约.md)，OpenAPI 草稿：[knowledge.openapi.yaml](../接口契约/openapi/knowledge.openapi.yaml)
- 智能问答：[智能问答-api契约.md](../接口契约/智能问答-api契约.md)，OpenAPI 草稿：[qa.openapi.yaml](../接口契约/openapi/qa.openapi.yaml)
- 报告生成：[报告生成-api契约.md](../接口契约/报告生成-api契约.md)，OpenAPI 草稿：[reports.openapi.yaml](../接口契约/openapi/reports.openapi.yaml)

拆分原则：

- 知识管理契约覆盖知识库、文档、处理任务、切片、检索、报告支撑材料和模型/解析配置。
- 智能问答契约覆盖会话、消息、流式问答、引用、意图路由、检索测试和问答配置。
- 报告生成契约覆盖模板、报告记录、大纲、章节生成、章节编辑、导出和下载。
- 用户认证、角色权限、文件上传下载由横向服务提供，三个业务契约只声明依赖关系。

## 11. 非功能需求

### 11.1 性能

- 文档解析、切片、向量化应异步执行，不阻塞用户上传后的页面操作。
- 智能问答应支持 SSE 流式输出，降低用户等待感。
- 报告内容生成应支持分章节流式输出，并展示章节级进度。
- 列表接口必须分页，避免大数据量拖慢页面。
- 检索接口必须支持 Top K 和阈值参数，避免返回过多低质量结果。

### 11.2 可用性

- 页面刷新后，登录态和会话历史应可恢复。
- 文档、问答、报告生成任务均需要清晰状态和失败原因。
- 空列表、失败状态、处理中状态需要明确提示。
- 删除类操作需要二次确认。

### 11.3 安全性

- 所有 API 请求必须认证。
- 管理功能必须鉴权。
- 下载原始文件和生成报告前必须校验权限。
- 模型密钥、对象存储 key、数据库错误、内部 URL 不得返回给前端。
- 重要操作后续可接入审计日志；首期不作为强制验收项。

### 11.4 可维护性

- 各服务只写自己拥有的数据表。
- 跨服务依赖通过 HTTP API 或明确契约完成。
- 运行时配置变更应可追踪，并尽量不要求重启服务。
- Qdrant collection 和 embedding 模型版本需要可追溯；embedding 维度变化时创建新版本 collection，并通过后台任务重建索引。

### 11.5 扩展性

- 支持新增知识库类型、文档类型、分段策略和检索策略。
- 支持新增报告类型和模板。
- 支持后续接入本地化模型、本地 OCR 后端和更复杂的权限模型。

## 12. 优先级建议

### P0：首期必须完成

- 用户认证与 RBAC 基础能力。
- 知识库、文档上传、解析、切片、向量化、检索。
- Qdrant、PostgreSQL、MinIO 基础集成。
- 智能问答 RAG 主流程和 SSE 流式输出。
- 两类固定报告的大纲、章节生成、编辑、DOCX 导出。
- 基础统计指标。

### P1：重要增强

- 引用溯源详情、文档下载权限校验。
- 检索体验测试。
- 模型运行时配置。
- 文档重处理。
- 模板管理和报告支撑材料管理。
- 报告记录重新导出。

### P2：加分项或后续迭代

- 本地模型部署。
- 模板可视化编辑器。
- 章节内容重新生成。
- 图片处理、按章节图表编号模式。
- Excel/表格数据分析意图。
- 全量审计日志、独立审计服务和细粒度组织权限。

## 13. 已确认决策与后续跟踪

详细来源见 [决策同步清单](./决策同步清单.md)。下面保留对开发排期影响最大的结论。

| 编号 | 结论 | 影响 |
| --- | --- | --- |
| Q1 | 认证统一为 Bearer Token/JWT。 | 影响前后端鉴权、SSE、下载接口、网关转发 |
| Q2 | API 统一前缀为 `/api/v1`。 | 影响所有契约和前端请求封装 |
| Q3 | 异步任务采用 Redis 队列 + PostgreSQL 持久化状态；任务最多自动重试 3 次，失败后进入 `failed` 并保留最近 10 次尝试摘要。 | 影响文档处理、报告生成、失败重试 |
| Q4 | OCR 和文档解析首期使用外部 HTTP 解析服务；通过 `parser.baseUrl`、`apiKey`、`timeoutSeconds`、`maxConcurrency` 配置。 | 影响文件处理链路和部署依赖 |
| Q5 | LLM、Embedding、Rerank 统一通过 `ai-gateway` 以 OpenAI-compatible API 调用。 | 影响配置字段、密钥管理、模型能力边界 |
| Q6 | 会话历史保存到服务端 PostgreSQL，前端只缓存 `sessionId` 等恢复信息。 | 影响 QA 会话/消息 API 和刷新恢复 |
| Q7 | 统计指标本期做；智能问答中的 Excel/表格数据分析意图本期不做。 | 影响智能问答意图路由和后续 API |
| Q8 | 报告支撑材料作为独立资源，复用 `file` service 和必要的 `knowledge` 检索能力。 | 影响知识管理与报告生成 API 边界 |
| Q9 | 首期采用角色级 RBAC，不做组织/电厂/专业多维权限。 | 影响知识库、文档、报告记录权限模型 |
| Q10 | 审计日志首期暂缓，后续可能独立成服务；首期只保留配置变更、任务失败、删除结果等基础排查字段。 | 影响安全与合规演进 |
| Q11 | Qdrant 使用版本化 collection；embedding 维度或模型族变化时创建新 collection 并重建索引。 | 影响知识库重处理和索引迁移 |
| Q12 | 报告大纲首期以数据库模板结构配置为权威，不从 DOCX 自动解析；`mode=ai` 不进入首期验收。 | 影响报告模板和大纲生成 |
| Q13 | DOCX 导出首期使用默认 style profile 和全局编号；`by_chapter` 与复杂图表编号保留为后续能力。 | 影响报告导出配置和验收范围 |
